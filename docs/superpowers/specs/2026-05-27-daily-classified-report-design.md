# 昨日分類日報設計稿

日期：2026-05-27
狀態：draft

## 1. 目標

新增一個介於 `dashboard / 今日監控` 與 `intelligence / 品牌決策` 之間的新產品層：

- 名稱：`昨日分類日報`
- 核心任務：讓品牌/行銷/公關人員在早會前 10 分鐘內完成「昨天一整天輿情」的分類匯報
- 第一版聚焦單一主品牌
  - 例如本案優先支援 `7-ELEVEN`
- 第一版交付形式：網頁頁面，可點回原文驗證內容

這一層不是值班盤，也不是月報。它回答的是：

- 昨天大家主要在聊什麼
- 哪些是風險，哪些是機會
- 各類別裡最值得匯報的訊號是什麼
- 有哪些原文與推文可作為匯報舉證

## 2. 非目標

第一版不做以下內容：

- 不做下載 PDF / 匯出簡報
- 不做推播或自動寄送
- 不做多品牌橫向對照
- 不改寫既有 `dashboard / 今日監控` 的語義
- 不把 `daily_snapshots` 當成日報主資料層

## 3. 使用者與使用情境

### 主要使用者

- 主品牌行銷人員
- 主品牌公關人員
- 需要在晨會口頭匯報昨日輿情的人

### 第一版使用情境

以 `7-ELEVEN` 行銷人員為例：

1. 早上打開系統，進入「昨日分類日報」
2. 先看一句總結與 6 大分類分布
3. 進入昨天最重要的 1-2 類，閱讀：
   - 重點觀察
   - 代表文章
   - 代表推文/留言摘錄
4. 在晨會中用系統內容回答：
   - 昨天大家主要討論哪些議題
   - 哪些屬於風險
   - 哪些屬於商品或行銷機會
   - 消費者的原話是什麼

此頁面若無法支撐上面 4 步，就視為失敗。

## 4. 產品邊界

本專案將維持三層產品結構：

- `dashboard / 今日監控`
  - 目的：值班監控、當下高風險辨識
  - 主體：signal/article + alert
- `daily report / 昨日分類日報`
  - 目的：昨天整日輿情分類匯報
  - 主體：固定分類 + 代表訊號 + 舉證摘錄
- `intelligence / 品牌決策`
  - 目的：30/90 天與月度策略觀察
  - 主體：event case / topic / monthly snapshot

硬限制：

- `昨日分類日報` 不得直接修改既有 `dashboard` 的數字與風險計算
- `daily_snapshots` 仍然只服務舊 dashboard 快照契約
- 新日報一律使用獨立資料層與獨立 API

## 5. 第一版固定分類

第一版固定 6 類，不開放自訂：

1. `食安 / 品質異常`
2. `商品反饋`
3. `服務體驗`
4. `價格 / CP 值`
5. `活動 / 聯名 / 行銷`
6. `會員 / 支付 / APP`

分類的目的是支撐晨會匯報，不是追求學術完美。第一版允許部份訊號落入 `其他/未分類`，但頁面不主推該類。

## 6. 頁面資訊架構

頁面為新頁：

- `daily-report.html`

### 6.1 頁首

- 日期切換：預設顯示昨天
- 主品牌選擇：第一版先固定主品牌，後續再擴充
- 返回入口：
  - `今日監控`
  - `品牌決策`

### 6.2 昨日一句話總結

頁面最上方顯示一段可口頭匯報的 opening：

- 範例：
  - `昨日 7-ELEVEN 輿情以商品反饋與價格討論為主，未見明顯食安危機，但聯名商品的價值感爭議升高。`

第一版可先用規則產出，不必一開始就用 LLM。

### 6.3 六大分類總覽

每類顯示：

- 昨日訊號數
- 正/中/負分布
- 是否有高風險訊號
- 類別排序

排序原則優先考慮：

- 訊號量
- 負面密度
- 是否含高分負面

### 6.4 分類詳情區塊

每個分類至少顯示：

- 分類名稱
- 昨日訊號數
- 正/中/負分布
- `重點觀察`
  - 2-4 句
  - 不是抽象形容詞，而是可直接拿去晨會講的句子
- 代表文章 3 則
  - 標題
  - 渠道
  - 情緒
  - 分數
  - 主題
  - 原文連結
- 代表推文 / 留言摘錄
  - 直接作為 `重點觀察` 的舉證，不另外拆成孤立模組

## 7. 內容生成原則

### 7.1 第一版先用規則聚合

第一版採 `方案 A：分類日報頁，純規則聚合`：

- 固定分類
- 固定代表文章數
- 固定摘錄數
- 固定句型的 `重點觀察`

原因：

- 先驗證分類與內容正確性
- 降低 token 成本
- 避免在分類不穩時過早引入長文本 AI 摘要

### 7.2 重點觀察應如何書寫

重點觀察需符合以下原則：

- 可直接拿去口頭匯報
- 要指出：
  - 昨天主要在談什麼
  - 消費者在意點是什麼
  - 是風險還是機會
- 後面應能被代表文章與代表推文支持

不接受的寫法：

- 純抽象大詞
- 只有情緒沒有內容
- 與下方文章證據對不起來

## 8. 資料模型

新增獨立日報層，避免污染既有 `daily_snapshots` 與 `intelligence` schema。

### 8.1 建議新表

#### `intel_daily_reports`

用途：存單日、單品牌的整體日報快照。

建議欄位：

- `report_date`
- `scope_type`：第一版固定 `brand`
- `scope_key`：例如 `7-ELEVEN`
- `snapshot_at`
- `headline_summary`
- `payload_json`

#### `intel_daily_report_sections`

用途：存每個分類區塊。

建議欄位：

- `report_date`
- `scope_key`
- `section_key`
- `section_label`
- `signal_count`
- `pos_count`
- `neu_count`
- `neg_count`
- `high_risk_count`
- `summary_text`
- `top_threads_json`
- `evidence_quotes_json`
- `payload_json`

### 8.2 不新增新原始資料表

原始訊號、分析、推文/留言仍共用現有資料表。

新日報層只做：

- 分類
- 聚合
- 摘要快照

## 9. 分類邏輯

日報分類不直接等於既有 `theme`。需新增一層穩定 taxonomy。

### 9.1 分類來源

每筆訊號分類可綜合使用：

- `channel`
- `title`
- `content`
- `theme`
- `sentiment`
- `score`
- `push_items / item analyses`

### 9.2 第一版分類優先順序

1. `食安 / 品質異常`
2. `商品反饋`
3. `服務體驗`
4. `價格 / CP 值`
5. `活動 / 聯名 / 行銷`
6. `會員 / 支付 / APP`

若一筆訊號同時命中多類，以優先順序決定主分類。

## 10. 日報建置流程

新增日報 builder job，例如：

- `run_daily_classified_report_capture()`

流程：

1. 讀取指定日期、指定主品牌的已分析訊號
2. 將訊號歸入 6 大分類
3. 每類挑選代表文章
4. 從推文/留言中挑代表摘錄
5. 產出每類 `重點觀察`
6. 寫入 `intel_daily_reports` 與 `intel_daily_report_sections`

### 10.1 執行時機

第一版建議：

- 不綁在每次 monitor 完成後立刻跑
- 改為獨立 daily job，預設產出昨天資料

理由：

- 日報是整日整理，不是即時監控
- 較符合晨會使用情境
- 可減少一日內重複計算與資料抖動

## 11. API 設計

新增獨立 API，不動既有 dashboard/intelligence 契約。

### 11.1 第一版端點

- `GET /daily-report`
  - 參數：
    - `date`
    - `scope_key`
  - 回傳：
    - 整體日報頁所需資料

第一版只做單一端點，減少前後端複雜度。若未來頁面需要 lazy load，再新增分區端點。

## 12. 與既有層的關係

### 12.1 與 dashboard 的關係

- `dashboard` 仍然偏高風險摘要
- `daily report` 是分類匯報
- 兩者可以共用原始資料，但不能共用聚合語義

### 12.2 與 monthly intelligence 的關係

- `daily report` 是日層級
- `monthly intelligence` 是長期層級
- 月報未來可引用日報分類結果作為素材，但第一版不耦合

## 14. 成功標準

第一版成功的標準不是功能多，而是：

1. 使用者能在 10 分鐘內完成昨天輿情報告準備
2. 每個分類都能用代表文章與推文舉證
3. 內容不會污染既有 dashboard 的風險語義
4. 主品牌行銷人員能直接說出：
   - 昨天主要討論什麼
   - 哪些是風險
   - 哪些可作為商品或行銷機會

若做不到以上 4 點，則此頁面仍屬低價值資訊牆，不應視為完成。
